# Cloud Streaming Setup Guide

This guide explains what values the firmware needs, where to get each one, how to run the local WebSocket receiver, and where to enter the values before flashing the ESP32.

The short version:

1. Start the local WebSocket receiver on your computer.
2. Find your computer's LAN IP address.
3. Put Wi-Fi, receiver URL, device ID, and token into `idf.py menuconfig`.
4. Build, flash, and monitor the board.
5. Press Record and confirm the receiver logs audio frames.

## What You Are Setting Up

The firmware can now stream recorded audio over Wi-Fi to a WebSocket receiver.

The ESP32 connects like this:

```text
ESP32 -> your Wi-Fi router -> your computer running the receiver
```

For local testing, the receiver is the FastAPI app in this repo:

```text
cloud.api.device_ws:app
```

The firmware sends:

- a `HELLO` JSON message first
- binary PCM audio frames while recording
- sequence numbers so the receiver can ACK received audio

## Values You Need

You need five values.

| Value | What it is | Example |
| --- | --- | --- |
| Wi-Fi SSID | The name of your Wi-Fi network | `MyHomeWifi` |
| Wi-Fi password | The password for that Wi-Fi network | `your-wifi-password` |
| Cloud WebSocket URL | Where the ESP32 sends audio | `ws://192.168.1.23:8000/v1/device/ws` |
| Device ID | A stable name for this board | `esp32s3-dev-01` |
| Collection token | A shared auth token used by firmware and receiver | `dev-token` |

Do not commit real Wi-Fi passwords or production tokens to Git.

## 1. Get Your Wi-Fi SSID

Your Wi-Fi SSID is the network name your computer or phone is connected to.

On macOS, you can usually see it from:

```sh
System Settings -> Wi-Fi
```

Use the exact network name.

Example:

```text
StarHome5G
```

## 2. Get Your Wi-Fi Password

Use the password for the same Wi-Fi network.

The ESP32 and your computer should be on the same Wi-Fi network for local receiver testing.

If your router has separate networks, prefer a normal 2.4 GHz or 5 GHz home LAN. Avoid guest networks if they block devices from talking to each other.

## 3. Choose a Collection Token

For local development, you make this up yourself.

Use something simple while testing:

```text
dev-token
```

The important rule is that the receiver and firmware must use the same token.

Receiver:

```sh
CLOUD_COLLECTION_TOKEN=dev-token
```

Firmware menuconfig:

```text
Collection token = dev-token
```

For production later, this should be a real secret token generated by your backend or deployment process.

## 4. Choose a Device ID

The device ID is not a password. It is just a stable name for the board.

Good local examples:

```text
esp32s3-dev-01
starspulse-watch-01
desk-device-01
```

Use letters, numbers, and dashes. Keep it short and recognizable.

## 5. Find Your Computer's LAN IP Address

The ESP32 needs to know how to reach your computer over Wi-Fi.

On macOS, run:

```sh
ipconfig getifaddr en0
```

That often prints something like:

```text
192.168.1.23
```

If that prints nothing, try:

```sh
ipconfig getifaddr en1
```

You can also check:

```sh
System Settings -> Wi-Fi -> Details -> IP address
```

Use the IP address from the Wi-Fi interface.

## 6. Build the Local WebSocket URL

For the local receiver, the URL format is:

```text
ws://YOUR_COMPUTER_IP:8000/v1/device/ws
```

If your computer IP is:

```text
192.168.1.23
```

Then the firmware WebSocket URL is:

```text
ws://192.168.1.23:8000/v1/device/ws
```

Use `ws://` for local testing.

Use `wss://` only when you have a real HTTPS/TLS server or reverse proxy deployed.

## 7. Start the Local WebSocket Receiver

From the repo root:

```sh
cd /Users/starspulse/myProject3
```

Create a Python virtual environment:

```sh
python3 -m venv .venv-cloud
```

Activate it:

```sh
. .venv-cloud/bin/activate
```

Install the receiver dependencies:

```sh
pip install -r cloud/requirements.txt
```

Start the receiver:

```sh
CLOUD_COLLECTION_TOKEN=dev-token CLOUD_AUDIO_DEBUG_WAV=1 \
uvicorn cloud.api.device_ws:app --host 0.0.0.0 --port 8000
```

Leave this terminal running.

When the ESP32 connects, this terminal should show WebSocket activity. When recording starts, it should receive audio frames and send ACKs.

## 8. Put the Values into ESP-IDF menuconfig

Open a new terminal and go to the repo:

```sh
cd /Users/starspulse/myProject3
```

Load ESP-IDF:

```sh
source /Users/starspulse/.espressif/v6.0.2/esp-idf/export.sh
```

Open menuconfig:

```sh
idf.py menuconfig
```

Go to:

```text
Project Runtime Configuration
```

Set:

```text
Wi-Fi station SSID
Wi-Fi station password
Cloud WebSocket URL
Device ID
Collection token
```

Example local setup:

```text
Wi-Fi station SSID: StarHome5G
Wi-Fi station password: your-wifi-password
Cloud WebSocket URL: ws://192.168.1.23:8000/v1/device/ws
Device ID: esp32s3-dev-01
Collection token: dev-token
```

Optional setting:

```text
Enable local WAV debug mirror
```

Leave this enabled while testing if you also want a local debug WAV written on the ESP32.

Save and exit menuconfig.

This writes your local values into `sdkconfig`.

## 9. Build the Firmware

```sh
idf.py -B build-codex2 build
```

If this passes, the firmware is buildable.

## 10. Find the ESP32 Serial Port

Plug in the board and run:

```sh
ls /dev/tty.*
```

Common port names look like:

```text
/dev/tty.usbserial-0001
/dev/tty.usbmodem1101
```

Use the one that appears when the ESP32 is plugged in.

## 11. Flash and Monitor

Replace the port with your actual port:

```sh
idf.py -B build-codex2 -p /dev/tty.usbserial-0001 flash monitor
```

If your port is `/dev/tty.usbmodem1101`, use:

```sh
idf.py -B build-codex2 -p /dev/tty.usbmodem1101 flash monitor
```

## 12. What Success Looks Like

In the ESP32 monitor, look for:

```text
Wi-Fi connected
got ip
WebSocket connected
HELLO queued
HELLO_ACK received; WebSocket session ready
AUDIO_ACK received seq=...
```

In the ESP32 monitor, these repeated lines should not appear after the first connection:

```text
The client has started
WebSocket start failed: ESP_FAIL
```

In the receiver terminal, look for connection and audio metadata logs. Opening `http://localhost:8000/` in a browser may show `GET /` or `GET /favicon.ico` with `404`; those are harmless because this app exposes a WebSocket endpoint, not an HTTP landing page.

When you press Record on the device:

- the ESP32 UI should show streaming or buffering status
- the receiver should receive binary audio frames
- the receiver should send `AUDIO_ACK`
- if `CLOUD_AUDIO_DEBUG_WAV=1`, the receiver should write a debug WAV file
- the receiver should not log `HELLO required before audio frames`
- the receiver should not log `wave.Error: # channels not specified`

## Troubleshooting

### The ESP32 does not connect to Wi-Fi

Check:

- SSID spelling
- Wi-Fi password
- whether the network allows new devices
- whether the board has enough signal

Then rebuild/flash after saving menuconfig.

### The ESP32 connects to Wi-Fi but not WebSocket

Check:

- the receiver terminal is still running
- the computer IP is correct
- the firmware URL uses your computer IP, not `localhost`
- the URL path is `/v1/device/ws`
- port `8000` matches the receiver command
- your computer firewall is not blocking incoming connections

Correct local URL shape:

```text
ws://192.168.1.23:8000/v1/device/ws
```

Wrong for ESP32 local testing:

```text
ws://localhost:8000/v1/device/ws
```

`localhost` from the ESP32 means the ESP32 itself, not your computer.

### The receiver rejects the connection

Check that the firmware token matches the receiver token.

Receiver:

```sh
CLOUD_COLLECTION_TOKEN=dev-token
```

Firmware:

```text
Collection token = dev-token
```

### The firmware says cloud upload is disabled

Cloud upload requires all of these to be set:

- Wi-Fi SSID
- Wi-Fi password
- Cloud WebSocket URL
- Collection token

If any are empty, the firmware will skip cloud audio upload.

### I changed menuconfig but nothing changed on the board

Run build and flash again:

```sh
idf.py -B build-codex2 build
idf.py -B build-codex2 -p YOUR_PORT flash monitor
```

## Quick Local Test Checklist

Use this checklist when testing:

```text
[ ] Computer and ESP32 are on the same Wi-Fi network
[ ] Receiver dependencies installed
[ ] Receiver running on port 8000
[ ] Computer LAN IP found
[ ] Firmware Cloud WebSocket URL uses ws://COMPUTER_IP:8000/v1/device/ws
[ ] Firmware Collection token matches CLOUD_COLLECTION_TOKEN
[ ] Firmware built successfully
[ ] Firmware flashed successfully
[ ] Monitor shows Wi-Fi connected
[ ] Receiver shows WebSocket connection
[ ] Monitor shows HELLO queued
[ ] Monitor shows HELLO_ACK received; WebSocket session ready
[ ] Monitor shows AUDIO_ACK received seq=... after recording starts
[ ] Recording sends audio frames
[ ] Receiver logs audio frame metadata
[ ] No repeated WebSocket start failed loop
[ ] No HELLO required before audio frames rejection during normal recording
[ ] No debug WAV wave.Error traceback
```
