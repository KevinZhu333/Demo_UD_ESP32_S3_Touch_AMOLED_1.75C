# Demo Spec · Always-on 可穿戴硬件 ·「开口循环 + 转发文章」全链路 Demo

> **本文是什么**：本 Demo 的**需求与约束契约（spec）**。它定义"**做什么 / 为什么做 / 面向谁 / 成功标准 / 范围边界 / 输入输出 / 行为规则 / 异常情况**"，**不定义"怎么实现"**（架构、模块拆分、技术选型、接口细节、数据库、算法实现等属于 `design.md`）。
> **怎么用**：开发 Agent（Claude Code / Codex 等）应**仅凭本文**产出 `design.md` 与后续实现，无需任何其它文档。本文是**自包含**的：所有术语在 §3 给出定义。
> **稳定性约定**：本文是**尽量不改的契约**。§8 行为规则（RULE-*）与 §9 异常处理是**冻结条款**；需要随测试调整的只有 §11 的 **可调参数（PARAM-*）的数值**，调数值不算改契约。
> **状态**：Spec v1 · **日期**：2026-06-18

## 1. 背景与要解决的问题（为什么做）

**产品一句话**：一台 always-on（整天在场采集）的可穿戴设备，在用户**最没有余力的时刻**，用一次"**双击 = 要 / 无视 = 算了**"的最小动作，替用户把"该做却做不到的小事"低摩擦地办掉；它**主动先递**，而不是等用户去问、去搜。

**本 Demo 要解决的问题**：在投入自研硬件与复杂能力之前，用现成器件搭一条**端到端全链路**，回答两个问题：

1. **链路是否可行**：从"采集/接收 → 处理 → 在用户空闲时主动递一句 → 用户 1-bit 确认 → 履约/播放 → （可选）外部续聊"这条链路，能否在真实条件下**端到端、尽量自动地**跑通。
2. **主动打扰是否被接受**：用户在被设备**主动**递一句（而非自己发起）时，是否愿意以 1-bit 回应、且**不把它当骚扰**。

> 这两个问题是后续一切的地基：链路不通或主动递不被接受，产品就不成立。

## 2. 目标与非目标

### 2.1 本 Demo 的目标（验收看这些，见 §10）
- **G1 链路可行性**：两条轨道（§4）各自端到端跑通，且**尽量全自动、无人工硬断**。
- **G2 主动递被接受**：真实用户在空闲时被主动推送后，达到可接受的"双击/不反感"比例。
- **G3 守住契约**：§8 的行为规则**零违背**。

### 2.2 明确的非目标（本轮不做、不验证，**不得**因"顺手"而做）
- **N1 不验证商业护城河**：不验证"按品味分发"和"带交易抽成（take-rate）的履约"。
- **N2 不验证个性化深度**：不要求"越用越懂你的品味"成立（本轮的学习只到"择时变准"，见 §3）。
- **N3 不做自研量产硬件**：仅用现成开发板与现成耳机/手机（见 §11）。
- **N4 不做"替别人/自己花钱"的履约**（送礼/订位/引荐等）：留待后续阶段。
- **N5 不追求"信息加工质量"**：摘要/音频/看板做到"可用"即可，**质量分不是成败标准**（见 RULE-11）。

## 3. 面向谁 & 术语表（自包含）

### 3.1 面向谁
- **Demo 的测试用户**：都市高负荷的知识工作者/管理者/创始人/专业服务者（忙、时间紧、说得出很多"待办/想看"，愿意尝试 AI 硬件、对隐私务实但在意知情同意）。本轮只需 **3–6 名**这样的用户做小范围试用。
- **本文的读者**：负责实现 Demo 的开发 Agent 与工程师。

### 3.2 术语表（全文使用这些定义）
- **always-on（整天在场采集）**：设备在佩戴期间持续采集（本 Demo 指持续录音、先在设备本地保存完整音频分段，并在网络条件允许时上传完整分段），不靠唤醒词、不靠用户按键启动。
- **1-bit 推送 / 1-bit 确认**：系统主动递出**一条**提议，用户只需付出"1 比特"的反馈：**双击 = Yes（要）**；**无视/超时 = No（不要）**；**用户开口说话或有来电/消息打断 = ABORT（此刻不合适，静默中止）**。
- **开口循环（open loop）/ action-item**：用户在日常言语中**说出口但未闭合**的待办意图，如"得跟财务核那笔款""slide 12 的数据要改""提醒供应商报价"。
- **封板**：把当天若干"开口循环"一次性收拢、写入一个任务后端（清单/笔记），从而让用户"心里那件事被放下"。
- **原位捕获**：在用户**已经在用的工具内**直接接走意图——本 Demo 指用户在**微信内**把文章/文档转发给服务号，而不是要求用户打开一个新 App。
- **空闲（idle）**：用户当前**适合被打扰**的状态。判定信号至少包括：**静止或非驾驶移动 + 当前没有进行中的对话 + 非勿扰时段**（具体判定逻辑属 design，本文只规定"必须只在空闲时主动推送"）。
- **冷状态**：用户情绪平稳、注意力不被强占的时刻（如通勤、睡前、起身离开工位）；与之相对的是"热状态"（交谈中、忙乱、情绪上头）。
- **信息看板**：基于一篇/一组内容生成的、结构化、可阅读的摘要视图，支持用户做高亮/备注。
- **音频对话**：基于主题生成的、可在耳机收听的口播/对谈式音频。
- **MCP 续聊**：把某主题的内容包装成一个标准化接口（MCP = 让外部 AI Agent 接入工具/数据的开放标准），使用户能在**外部 Agent**（如 ChatGPT/Claude 等）里就该主题继续讨论。
- **time-to-dark**：一次 1-bit 交互从屏幕亮起到熄灭所用时间；本 Demo 把"越短越好"作为屏幕的目标（与"停留时长"相反）。
- **声纹门控**：尽量只保留"用户本人"的语音、过滤他人语音的机制（本 Demo 为"尽力而为"项）。
- **take-rate（交易抽成）**：平台在用户成交时抽取的佣金。**本 Demo 不涉及**，仅在术语上说明，便于理解非目标 N1。

## 4. 范围边界（In / Out of Scope）

### 4.1 范围内：两条轨道
- **轨道 A · 转发文章（内容/钩子）**：用户经微信服务号转发文章/文档 → 系统处理 → 生成"音频对话 + 信息看板" → 空闲时 1-bit 主动推送 →（Yes 后）耳机播音频、手机出看板 →（可选）经 MCP 在外部 Agent 续聊。
- **轨道 B · 开口循环（核心可行性验证）**：设备持续本地录音并上传完整音频分段 → 系统抽取"开口循环/action-items" → 在"起身离开/关机"触发点 1-bit 主动推送 →（Yes 后）自动把开口循环**封板写入任务后端**。

### 4.2 范围外（本轮不做）
- 任何"替用户向他人/商家发起的动作"（发消息给某人、下单、订位、送礼、引荐）。
- 自研/量产硬件；端侧大模型推理；端侧"原文不出端"的隐私架构（本轮允许原文上云，见 §11/§12 约束与技术债）。
- 真实的偏好图谱与"越用越懂品味"的学习闭环。
- 多轮语音对话助理（设备端**不做**对话；多轮讨论只发生在外部 Agent 上，见 RULE-2）。

## 5. 系统输入 / 输出（黑盒 I/O 契约）

> 只规定"必须接收/产出什么信息"，不规定格式/协议/编码（属 design）。

### 5.1 输入
- **IN-1 转发文档**：用户经微信服务号发来的**文章链接或文件**。系统必须能处理至少：网页链接、PDF、纯文本/Markdown 三类。
- **IN-2 持续音频分段**：设备连续采集、保存并在网络可用时上传的完整用户语音分段，用于抽取开口循环。
- **IN-3 用户 1-bit 反馈**：来自设备的 `Yes`（双击）/ `No`（无视或超时）/ `ABORT`（被打断）三态之一。
- **IN-4 情境信号**：用于判定"空闲"与触发时机，至少包括：运动/驾驶状态、是否对话中、时间（时段）；位置/日历为可选增强。

### 5.2 输出
- **OUT-1 1-bit 推送卡**（设备 1.75″ 屏）：一句话提议，一次只呈现**一张**；隐含语义为双击=Yes / 无视=No。
- **OUT-2 音频对话**（蓝牙耳机）：轨道 A 在用户接受主题后按需生成的 10–15 分钟可听内容，目标时长由 `PARAM-AUDIO_LEN` 控制。
- **OUT-3 信息看板**（手机）：轨道 A 的结构化摘要视图，支持用户高亮/备注。
- **OUT-4 任务后端写入**（轨道 B，Yes 后）：把封板的开口循环写入一个任务清单/笔记后端。
- **OUT-5 MCP 话题接口**：把主题/内容包装为 MCP 供外部 Agent 续聊；**禁止**经此接口暴露原始录音或逐字转写（见 RULE-6）。

## 6. 功能需求（FR，逐条可测）

> 句式：系统**必须**……。每条都应能被一个明确的测试验证。

### 6.1 轨道 A · 转发文章
- **FR-A1** 系统必须能接收用户经微信服务号转发的文章/文档（IN-1），并完成清洗、格式标准化与存储。
- **FR-A2** 系统必须从已存内容中**提取主题/要点**。
- **FR-A3** 系统必须基于主题生成**信息看板（OUT-3）**；音频对话（OUT-2）只在用户接受主题后，按 `PARAM-AUDIO_LEN` 目标时长生成。主题提取主路径可使用 NotebookLM，但必须经 Provider 抽象与唯一出境点；NotebookLM 不可用时允许 internal LLM degraded fallback 产出可推送 topic/board，fallback 不承诺跨文档 N:M 等价。
- **FR-A4** 系统必须在用户**空闲时（§3 idle）**以 **1-bit 推送（OUT-1）**主动提议某个主题（例："想了解《{主题}》吗？"），**不得**要求用户主动去查找或打开；该卡不表示音频已就绪。
- **FR-A5** 当用户在 FR-A7 播放提议中回应 `Yes`：系统必须播放音频对话（耳机）并在手机呈现信息看板；用户可在看板上做**高亮/备注**，且这些高亮/备注**首先属于用户自己**（可被用户取用），偏好信号只是副产品（见 RULE-10）。
- **FR-A6** 系统必须能把某主题的内容包装为 **MCP 接口（OUT-5）**，使用户可在外部 Agent 就该主题继续讨论。
- **FR-A7** 当用户接受 FR-A4 的主题后，系统必须持久化音频生成请求；音频就绪后，系统必须在新的空闲窗口以 **1-bit 推送（OUT-1）**主动提议播放该音频。

### 6.2 轨道 B · 开口循环
- **FR-B1** 设备必须能 always-on 持续录音，保存本地完整音频分段，并在可用网络下可靠上传完整分段（IN-2）。
- **FR-B2** 系统必须从语音转写中**抽取开口循环/action-items**（如"得跟进 X""别忘了改 Y"）。
- **FR-B3** 系统必须在**"起身离开/关机"类冷状态触发点**，以 **1-bit 推送**提议把当天开口循环**封板**（例："今天 4 个开口循环，封板到明早第一格？"）。
- **FR-B4** 当用户回应 `Yes`：系统必须**自动**把这些开口循环**写入任务后端（OUT-4）**，无需用户再做任何编辑或操作。
- **FR-B5** 轨道 B 的捕获与履约**必须做到全自动**（不依赖人工兜底）——这是本 Demo"链路可行性"的关键观测点。

### 6.3 设备与交互（两轨共用）
- **FR-D1** 设备屏一次只显示**一张** 1-bit 卡：双击 = `Yes`；超时无操作 = `No`；用户开口或来电/消息 = `ABORT`。
- **FR-D2** 任何一次 1-bit 交互结束后，屏必须**尽快熄灭**（追求小的 time-to-dark）；**禁止**自动加载下一条、**禁止**任何"继续看/猜你想看"的延展。
- **FR-D3** 设备**禁止**呈现信息流、未读红点、可下滑历史列表或 App 启动器式网格（见 RULE-3、RULE-8）。

### 6.4 通用
- **FR-G1** 系统**只在空闲时（§3 idle）**主动推送；**禁止**在用户对话中、驾驶中、勿扰时段打断。
- **FR-G2** 系统必须对主动推送做**全局合并节流**：Track-A 主题提议、Track-A 播放提议与 Track-B 封板提议共用同一 `PARAM-COOLDOWN` 与 `PARAM-DAILY_CAP`；两次推送间隔不低于 `PARAM-COOLDOWN`，每日主动推送数不超过 `PARAM-DAILY_CAP`。
- **FR-G3** 系统必须记录每次反馈（`Yes`/`No`/`ABORT`）并据此调整**后续推送时机**；其中 `ABORT` **只**用于调整"择时"，**不得**被当作"用户不喜欢该内容"（见 §9 EDGE-2）。
- **FR-G4** 用户必须能**随时停止采集**并**删除**自己的数据（见 RULE-7、§11 隐私约束）。

## 7. 用户旅程（端到端示例 · 验收走查脚本）

> 用两条具体旅程把 §6 的需求串成可演示的样子；它既是说明，也是验收时的走查脚本（每步标注对应需求/边界条款）。

**旅程一 · 转发文章（轨道 A）**
1. 用户在微信里把一篇"想看没空看"的长文转发给服务号。〔IN-1 / FR-A1〕
2. 系统清洗存储该文，提取主题与要点，生成 topic-scoped 信息看板；同一主题可来自多篇文档，一篇文档也可拆成多个主题。〔FR-A2 / FR-A3 / EDGE-8〕
3. 当天通勤途中（静止、未在对话、非勿扰 = 空闲），设备屏轻量浮出一张主题卡："想了解《XX》吗？"〔FR-A4 / FR-G1〕
4. 用户**双击**接受主题：系统持久化 `AUDIO_REQUESTED`，按 `PARAM-AUDIO_LEN` 目标时长生成音频；该请求在进程崩溃、设备离线、网络中断后仍可恢复补完。〔FR-A7 / EDGE-6〕
5. 音频就绪后，系统在另一个空闲且未触达全局节流/日上限的窗口再推一张播放卡："音频好了，现在听？"〔FR-A7 / FR-G2〕
6. 用户**双击**：耳机开始播音频，手机同时呈现信息看板，用户顺手高亮两句（高亮归用户自己）。〔FR-A5 / RULE-10〕
7. 确认后屏很快熄灭；用户晚些可在外部 Agent 里经 MCP 接口就该主题继续讨论。〔FR-D2 / AC-4 / FR-A6 / OUT-5 / RULE-6〕
> 若用户**无视主题卡**：记为 No，卡静默消失、不重复弹。若用户**无视播放卡**：表示"现在不听"，音频候选可在 TTL、全局节流和日上限约束内有限重议。〔EDGE-1 / EDGE-3〕

**旅程二 · 开口循环（轨道 B）**
1. 工作日里用户在不同对话中说出几个开口循环："得跟财务核那笔款""slide 12 要改""提醒供应商报价"。设备 always-on 本地录音并后台上传完整音频分段。〔IN-2 / FR-B1〕
2. 系统从转写中抽取这 3 条 action-items，去重归并。〔FR-B2 / EDGE-8〕
3. 傍晚用户起身离开工位（"关机"触发点），设备屏出一张卡："今天 3 个开口循环，封板到明早第一格？"〔FR-B3〕
4. 用户**双击**：系统**自动**把 3 条写入用户的任务后端，用户脑子清空、安心下班（全程无人工）。〔FR-B4 / FR-B5〕
> 若推送那一刻用户正巧开口说话：记为 ABORT，静默中止，仅调整下次择时、不改内容判断。〔EDGE-2〕

## 8. 行为规则（契约红线 · 冻结条款，开发中不可违背）

> 这些是定义"产品是什么"的边界。**即使为了 demo 方便也不能破**；破其一，demo 验证的就是另一个产品。每条都给出可检验的判定。

- **RULE-1 主动先递、禁止"让用户问"**：系统必须主动递出建议；**禁止**提供搜索框、问答框、"查我的记忆"等任何要求用户主动发起查询的入口。〔判定：产品中不存在用户发起检索的交互。〕
- **RULE-2 只收 1-bit、不追问、不做对话助理**：一次提议只接受 `Yes/No/ABORT`；**禁止**"有空吗"之类预问、**禁止**多轮追问、**禁止**在设备端做对话式助理（多轮讨论只允许发生在外部 Agent 上）。Track-A 主题提议与播放提议是两次独立的 1-bit 卡，各自等待 idle、各自过全局节流，不在设备端串成预问或会话态。〔判定：设备端不存在多轮会话。〕
- **RULE-3 阅后即焚、无信息流**：**禁止** feed、红点、未读角标、可下滑历史；屏一次只承载当前一件事，关完即灭。〔判定：界面无历史列表/红点。〕
- **RULE-4 屏以 time-to-dark 为目标、绝不追求停留时长**：**禁止**任何以"延长用户停留"为目的的机制（自动连播、无限流、诱导停留）。〔判定：无自动播放下一条、无无限滚动。〕
- **RULE-5 高风险不自动执行、只做低风险可逆动作**：任何对外动作必须在用户 `Yes` **之后**才发生；本 Demo 的履约**仅限**"写入用户自己的任务后端"这类**低风险、可逆**动作；**禁止**任何向他人/商家发起的发送/下单/转账/订位，**禁止**触碰医疗/金融/支付。〔判定：无未经 `Yes` 的对外动作；无对外发送/交易。〕
- **RULE-6 对外只出加工结果、绝不出原始素材**：任何第三方/MCP 接口**永不**输出原始录音或逐字转写，只输出主题/内容层的加工结果。〔判定：MCP/对外接口的输出中不含原始音频与逐字稿。〕
- **RULE-7 知情同意 + 只处理本人**：always-on 录音必须有用户**显式知情同意**；尽量只保留用户本人语音；录到的他人语音**不分析、不外传、可删除**；用户可随时停止与删除。〔判定：有同意流程；提供停止与删除能力。〕
- **RULE-8 不替代手机、不做主屏/启动器**：设备是叠加在既有工具之上的低摩擦层；**禁止**把设备屏做成通用 App 启动器或主屏。〔判定：屏不承载通用 App 网格。〕
- **RULE-9 去道德化、不说教**：推送话术只"递一个选项"，**禁止**催促、规训、评判（如"你又拖延了""该早睡了"）。〔判定：话术库中无管教/评判式表达。〕
- **RULE-10 打标自利、非捐赠**：让用户记笔记/高亮是**为用户自己**（可导出、可取用），**禁止**"帮我们改进/帮我们打分"式的利他标注。〔判定：笔记对用户自身可用；无"为我们标注"提示。〕
- **RULE-11 不以信息加工质量为成败标准**：摘要/音频/看板质量做到"可用"即可；**禁止**把"质量分"当作 demo 成败或核心指标。〔判定：验收指标（§10）中不含"加工质量评分"。〕
- **RULE-12 轨道 B 只当钩子、不被当护城河**：轨道 B（开口循环）是获客/可行性载体；它跑通**只证明**"链路可行 + 主动递被接受"，**不得**被表述/扩展为产品的差异化或利润中心。〔判定：文档/界面不把开口循环宣称为护城河。〕

## 9. 异常与边界情况（必须有明确行为）

| 编号 | 情况 | 期望行为 |
|---|---|---|
| **EDGE-1** | 用户对 1-bit 推送**无视/超时** | 记为 `No`；主题提议静默消失、不重复弹、不追问；播放提议的 `No/timeout` 表示"现在不听"，可受全局节流、日上限和 TTL 有限重议；按 FR-G3 仅微调后续择时 |
| **EDGE-2** | 推送瞬间用户**开口说话 / 来电 / 来消息** | 记为 `ABORT`（静默中止）；**只**调"择时"、**不**调"内容偏好"；稍后合适时机可再议 |
| **EDGE-3** | 一段时间内**找不到空闲窗口** | 宁可**不推送**，也不在热状态打断；候选意图可衰减/过期，不堆积成"待办债"；Track-A `audio_play` 可跨一个业务日边界等待，但仍受候选 TTL 绝对上限约束 |
| **EDGE-4** | 录音中**混入他人语音** | 按 RULE-7：不分析、不外传、可删除；尽力声纹门控；他人内容不得进入对外输出 |
| **EDGE-5** | **主题提取/音频生成等外部能力失败或超时** | 该条意图**静默丢弃或排队重试**，**禁止**向用户报错弹窗或要求用户处理；不得阻塞其它链路；主题提取可降级到 internal LLM 产出 topic/board，音频最终失败必须进入 fallback TTS / board-only / discarded 等明确终态 |
| **EDGE-6** | **网络中断 / 设备离线 / 重连** | 设备本地缓冲，恢复后续传；不得丢失"已确认（Yes）但未完成"的履约，恢复后补完；Track-A 主题接受后必须以 `AUDIO_REQUESTED`/audio job 检查点满足恢复补完 |
| **EDGE-7** | 抽取出的开口循环为**空**（当天没有可封板项） | 不推送"封板"提议（不为推送而推送） |
| **EDGE-8** | **重复/近似**意图（同一件事被多次说到） | 去重合并，不重复推送同一件事；Track-A topic 候选生成前也必须做跨文档/近似主题去重 |
| **EDGE-9** | 用户**撤回同意 / 请求删除数据** | 立即停止采集；按请求删除对应数据；删除须可被用户确认 |
| **EDGE-10** | 转发内容**无法解析**（格式不支持/抓取失败） | 静默跳过或给用户一条**非打扰式**的可选告知；不得反复打扰、不得阻塞 |
| **EDGE-11** | 用户**长时间未交互 / 屏意外常亮** | 屏按 time-to-dark 自动熄灭；不得停留 |

## 10. 成功标准与验收（可度量）

> Demo 被判为"成功验证了可行性"，须**同时**满足以下，且阈值数值见 §11（PARAM-*），可在不改本契约结构的前提下校准。

- **AC-1 链路贯通**：轨道 A、轨道 B 各至少一条意图**端到端跑通**。轨道 A 判定为「接收 → 主题/看板 → 主题推送 → 接受 → `AUDIO_REQUESTED` → 音频就绪 → 播放推送 → 播放」；Track-A `AUDIO_REQUESTED` 崩溃恢复补完计入 EDGE-6 recovery。轨道 B 因两端自动，贯通率应 ≥ `PARAM-LINK_THROUGH`。
- **AC-2 主动递被接受**：真人小测中，空闲主动推送的"双击/不反感"比例 ≥ `PARAM-ACCEPT_TARGET`；Track-A 主题提议、Track-A 播放提议与 Track-B 封板提议都进入主分母，并按 `push_kind` 分层展示。
- **AC-3 契约零破例**：§8 的 12 条 RULE 在整个 demo 期间**零违背**（尤其 RULE-5 不自动执行、RULE-6 不外泄原文、RULE-9 不说教、RULE-12 不把轨道 B 当护城河）。
- **AC-4 屏纪律**：典型 1-bit 交互的 time-to-dark ≤ `PARAM-TTD`；无信息流/红点/历史（FR-D2/D3）。
- **AC-5 技术债已记账**：§12 的每项妥协都在实现中**显式标注**，未被当作"已实现的最终方案"。

> **不作为验收指标**：摘要/音频/看板的质量分、界面美观度（见 RULE-11）。

## 11. 约束条件（给定前提 + 可调参数）

### 11.1 固定约束（design 与实现必须接受为前提）
- **CON-1 硬件给定**：设备侧使用现成 **ESP32-S3 + 1.75″ AMOLED 开发板**（承担"持续录音分段上传"与"1-bit 推送+双击确认"）；音频经**现成蓝牙耳机**；深读看板在**用户手机**。本轮不引入其它自研硬件。
- **CON-2 采集入口给定**：轨道 A 的"原位捕获"必须经**微信服务号**通道（须遵守其合规与文件类型限制；不得使用个人号自动化等封号灰区手段）。
- **CON-3 外部能力可用但可热插拔**：主题提取/音频生成等可直接使用现成第三方能力（如 NotebookLM）；但**禁止**把它写死为不可替换的核心（见 DEBT）。
- **CON-4 对外接口形态**：续聊能力必须以 **MCP** 形态对外（OUT-5），并受 RULE-6 约束。
- **CON-5 隐私笼子（录音上传的前提，缺一不可）**：① 显式知情同意；② 尽量只录本人、他人语音不分析不外传可删除；③ demo 数据隔离存储、用户可一键删除、demo 结束即清理；④ 对外只出加工结果不出原始素材；⑤ "原文上云"仅限 demo 阶段（见 DEBT-1）。

### 11.2 可调参数（仅数值可改，改数值不算改契约）
| 参数 | 含义 | 默认值（待真人小测校准） |
|---|---|---|
| **PARAM-RESP_TIMEOUT** | 1-bit 等待用户操作的超时（超时即记 `No`） | 20 秒 |
| **PARAM-COOLDOWN** | 两次主动推送的最小间隔 | ≥ 60 分钟 |
| **PARAM-DAILY_CAP** | 每日主动推送次数上限 | ≤ 6 次 |
| **PARAM-IDLE_HOLD** | 判定"空闲"所需的持续静止/无对话时长 | ≥ 60 秒 |
| **PARAM-TTD** | 一次 1-bit 交互的 time-to-dark 上限 | ≤ 10 秒 |
| **PARAM-AUDIO_LEN** | 轨道 A 音频生成目标时长 | 12 分钟（合法区间 5–20 分钟；demo 目标 10–15 分钟） |
| **PARAM-LINK_THROUGH** | 链路贯通率目标 | ≥ 95% |
| **PARAM-ACCEPT_TARGET** | 主动递接受率目标 | ≥ 50% |

## 12. 技术债声明（本 Demo 故意欠下；design/实现须知其为"临时"）

- **DEBT-1 原文/音频上云**：本轮以"持续录音分段上传 + 云端处理原文"替代"端侧编码 + 密文存储 + 只吐裁决"的目标隐私架构。仅限 demo，受 CON-5 约束；进入真实产品前须偿还为端侧方案。
- **DEBT-2 依赖第三方加工能力**（如 NotebookLM）：仅作"借引擎跑通链路"，须保持可替换。
- **DEBT-3 空闲识别为近似实现**：本轮可用启发式（时段/静止/无对话）近似"空闲"，非真实端侧空闲质量模型。
- **DEBT-4 偏好/个性化为极简**：本轮只做到"择时变准"，不做真实偏好图谱与"越用越懂品味"。
- **DEBT-5 护城河场景不在本轮**：带 take-rate 的品味分发（如"替你把人情/犒劳花出去"）整轮不做，留待后续阶段。
- **DEBT-6 MCP 暴露粒度**：本轮对外暴露的是主题/内容层（受 RULE-6 约束），尚非"只吐偏好裁决"的最终形态。

## 13. `design.md` 应当回答的问题（便于据本 spec 直接推进设计，不需额外信息）

> 以下是 design 阶段要解决的"怎么做"。本 spec 已给出全部"做什么/约束"，design 只需在不违反 §8 与 §11 的前提下作答：

1. **端到端架构与数据流**：设备 ↔ 微信服务号 ↔ 云端 ↔ 外部能力 ↔ MCP 的整体结构与各段数据流。
2. **设备侧固件**：录音采集与上传、1-bit 卡的显示/双击/超时/打断、屏的 time-to-dark 实现。
3. **云端处理管线**：文档清洗/转格式/存储；主题提取；音频对话与信息看板生成；失败重试与丢弃策略（满足 EDGE-5）。
4. **开口循环抽取**：从转写中抽取 action-items 的方法、去重合并（满足 FR-B2、EDGE-8）。
5. **空闲识别与触发**：如何用 IN-4 判定 idle 与"起身离开/关机"触发点（满足 FR-G1、FR-B3、EDGE-3）。
6. **任务后端写入**：封板写入哪个后端、字段与幂等（满足 FR-B4、EDGE-6）。
7. **反馈回写与择时调整**：`Yes/No/ABORT` 如何回写、如何只调择时不调内容（满足 FR-G3、EDGE-2）。
8. **MCP 接口设计**：暴露什么、如何确保不泄露原文（满足 OUT-5、RULE-6）。
9. **隐私实现**：同意流程、声纹门控、数据隔离与删除、限期清理（满足 RULE-7、CON-5、EDGE-9）。
10. **关键参数落位**：§11 各 PARAM 的取值与可配置方式。
11. **可观测性**：如何采集 §10 的验收指标（链路贯通率、接受率、time-to-dark、契约破例监测）。
