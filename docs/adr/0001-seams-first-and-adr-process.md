# ADR-0001 · 接缝先行 + ADR 制度

- **状态**: Accepted
- **日期**: 2026-06-23
- **决策者**: A（架构师）
- **相关**: management.md §0/§2/§7.3/§8.3；spec §13；design §0.3
- **影响的 spec ID**: 全部接缝相关 ID（K1–K11 覆盖的 IN/OUT/FR/RULE/PARAM/CON/AC）

## 背景

3 人并行开发同一全链路系统，最大的协作风险是「接口漂移」与「互相等待」。design.md §0.3 已钉死跨切片不变量的唯一执行点；本 ADR 把它落成团队流程：先冻结接缝，再并行实现。

## 决策

1. **接缝先行**：里程碑 M0 唯一目标 = 把对接契约 K1–K11 以「类型/Schema/桩 + 存在性测试」合入 `main`。M0 完成前不开始依赖接缝的实现 PR。
2. **协议单源**：Device Protocol 以 `protocol/*.schema.json` 为唯一事实源，CI 生成 pydantic（`cloud/common`）与 TS（`simulator`）两端类型，禁止手写对齐（钩子 H4）。
3. **ADR 制度（demo 右尺度）**：ADR 记录**对接缝/红线的契约性变更**与**有争议的跨切片决策**，**不强制 M0 冻结时每条接缝各写一份**（11 条接缝初始冻结由本 ADR 一条总述）。改接缝必带 ADR + @architect 评审。
4. **文档-代码同步靠钩子**：H1–H5（management.md §7）在 pre-commit + CI 强制，spec/design 与代码不可悄悄脱节。

## 取舍

- **取**：消除接口漂移与等待；红线变更可追溯；新成员可凭 ADR 理解「为什么」。
- **舍**：M0 有前期开销、接缝变更有流程成本。对 3 人并行而言，这点成本远低于集成期返工。

## 后果

- M0 退出条件即「K1–K11 桩合入 + 单元构建入口 + CI 必需检查全绿」（management.md §6.2）。
- 后续每条接缝变更新增一份 ADR；本文件是模板与首条记录。
